## Filtering  

:::note Update Before Running
Change `NAME_PREFIX%` in the `WHERE` clause to match your program name prefix. Use `'%'` to return all scheduled requests with no name filter.
:::

| Filter | Purpose |
|---|---|
| `r.phase_code = 'P'` | Pending (scheduled in manager), not already running or completed |
| `NVL(c.date2, SYSDATE + 1) > SYSDATE` | Release class is still active/valid |
| `c.class_type IS NOT NULL` | Request has a defined schedule |
| `UPPER(NVL(...)) LIKE 'NAME_PREFIX%'` | Scope to your custom programs; falls back to program name if description is null |

## `classInfo` Reference

`classInfo` is a **VARCHAR2 bit string** stored by EBS to encode schedule patterns. It is always processed as character data â€” never cast to a number.

| Bit Position | Length | Meaning | Example |
|---|---|---|---|
| 1â€“31 | 31 | Specific days of month | `1` at position 5 = 5th of month |
| 32 | 1 | Last day of month | `1` = enabled |
| 33â€“39 | 7 | Days of week `Su Mo Tu We Th Fr Sa` | `0111110` = every weekday |
| 40â€“44 | 5 | Week of month `1stâ€“5th` | `10100` = 1st and 3rd week |
| 45â€“56 | 12 | Months `Janâ€“Dec` | `111111111111` = every month |

:::caution Common Pitfall: "First day shows as 16"
This happens when code numerically casts the bit string â€” leading zeros are dropped, shifting all positions. This query uses `SUBSTR(...) = '1'` and `INSTR(..., '1')` on character data exclusively, so positions are always accurate.
:::

## Schedule Decoding 

<Tabs>
  <TabItem value="periodic" label="Periodic (P)" default>

`classInfo` format: `interval:unit:mode`

| Token | Values | Meaning |
|---|---|---|
| `interval` | Number | How often |
| `unit` | `N` `H` `D` `M` | Minutes, Hours, Days, Months |
| `mode` | `S` `C` | From start of prior run / From completion of prior run |

**Output example:** `Repeat every 4 hours from the completion of the prior run`

  </TabItem>
  <TabItem value="specific" label="Specific Days (S)">

Assembled in this order â€” each piece is optional and concatenated:

1. **Explicit dates** â€” if any of bits 1â€“31 are set â†’ `Dates: 1, 5, 20`
2. **Last day** â€” if bit 32 set â†’ `Last day of month`
3. **Day of week** (shortcutted first):
   - `1111111` â†’ `Daily`
   - `0111110` â†’ `Every weekday`
   - Any other pattern â†’ `Days of week: Mo Tu Fr`
4. **Week of month** â€” only if some but not all of bits 40â€“44 are set â†’ `Weeks: 1st 3rd`; omitted if `00000` (all weeks) or `11111` (all weeks)
5. **Months** â€” if bits 45â€“56 = `111111111111` â†’ `, Every Month`; if partial â†’ `in Jan Mar`; if none â†’ defaults to `, Every Month`

**Output example:** `Days of week: Mo Tu We Th Fr, Every Month`

### Performance Notes

- The `base` CTE applies all `WHERE` filters early, minimizing rows passed to `dates` and the final `SELECT`.
- `dates` uses a single `CONNECT BY LEVEL <= 31` cross join with `LISTAGG` â€” no row-by-row processing or PL/SQL.
- All schedule decoding uses `SUBSTR`/`INSTR`/`DECODE` on character data â€” no type conversions or scalar subqueries.
- `FND_LOOKUP_VALUES` is joined once with enablement and date filters inline, not as a subquery per row.

### Extensibility

**Locale-aware day/month names:** Replace hard-coded tokens (`'Mo '`, `'Jan '`) with a small lookup CTE joined by position index and `LISTAGG`-ed.

**Duplicate status meanings:** If `FND_LOOKUP_VALUES` returns multiple rows for a `status_code` (e.g., entries like `@Normal` alongside `Normal`), wrap the join target in a subquery using `ROW_NUMBER() OVER (PARTITION BY lookup_code ORDER BY ...)` to select one canonical meaning per code. Alternatively, add `AND flvStatus.meaning NOT LIKE '@%'` as a quick filter if the `@` prefix pattern is consistent in your instance.
