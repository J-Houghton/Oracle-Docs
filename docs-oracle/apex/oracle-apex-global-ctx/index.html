Request-Scoped Global Context in APEX 23.1 (Oracle 19c+)

Related

See SQL Macros vs Global Context for a comparison of approaches.

Apply per-user, per-page filters across pooled DB sessions and Ajax requests using a Global Application Context keyed by a canonical client_id. Values are read via SYS_CONTEXT, set in an APEX request initializer, and cleared on cleanup.

How It Works​

A global context stores key/value pairs in the SGA scoped by CLIENT_IDENTIFIER.SYS_CONTEXT reads values; DBMS_SESSION.SET_CONTEXT writes them; DBMS_SESSION.CLEAR_CONTEXT removes them. APEX runs Initialization and Cleanup PL/SQL at the start and end of every request — including Ajax — making this work consistently across pooled sessions.

Ajax Gotcha

Before Header and After Regions page processes do not fire on Ajax requests (filter/sort/partial refresh). For Ajax or mixed flows, always use Shared Components → Security Attributes → Database Session.

Attribute name must match exactly

The package uppercases attribute names. Make sure the string passed in when setting the attribute matches the string used in SYS_CONTEXT. Example:

  • Setter uses 'P_SALARY_IN'
  • Reader uses SYS_CONTEXT('TEST_APP_CTX','P_SALARY_IN')

Prerequisites​

Create in this order
  1. Create the context first, bound to the trusted package.
  2. Create the package that calls SET_CONTEXT / CLEAR_CONTEXT.
  3. (Optional) Grant EXECUTE on the package to the APEX parsing schema.
  4. Use a consistent client_id pattern: :APP_USER || ':AppName:PageId'.

Only the package named in CREATE CONTEXT ... USING can set attributes.

Database Setup​

  • 1. Create Context
  • 2. Create Package
CREATE OR REPLACE CONTEXT TEST_APP_CTX
    USING test_app_ctx_helper_pkg
    ACCESSED GLOBALLY;

ACCESSED GLOBALLY stores attributes in the SGA and exposes them across sessions via CLIENT_IDENTIFIER.

CREATE OR REPLACE PACKAGE xxx_app_ctx_helper_pkg AUTHID DEFINER AS

    PROCEDURE set_parameter_value(
        p_client_id IN VARCHAR2, 
        p_attr      IN VARCHAR2, 
        p_val       IN VARCHAR2
    );

    PROCEDURE clear_ctx(p_client_id IN VARCHAR2);

END xxx_app_ctx_helper_pkg;
/
CREATE OR REPLACE PACKAGE BODY xxx_app_ctx_helper_pkg AS
    PROCEDURE set_parameter_value(p_client_id IN VARCHAR2, p_attr IN VARCHAR2, p_val IN VARCHAR2) IS
    BEGIN
        DBMS_SESSION.set_context(
            namespace  => 'TEST_APP_CTX',
            attribute  => UPPER(p_attr),
            value      => p_val,
            username   => null,
            client_id  => p_client_id
        );
    END;
    PROCEDURE clear_ctx(p_client_id IN VARCHAR2) IS
    BEGIN
        DBMS_SESSION.clear_context(
            namespace  => 'TEST_APP_CTX',
            client_id  => p_client_id,
            attribute  => null  -- null clears ALL attributes for this client_id
        );
    END;
END xxx_app_ctx_helper_pkg;
/

APEX Integration​

Configure once per application at: Shared Components → Security Attributes → Database Session.

  • Initialization PL/SQL
  • Cleanup PL/SQL
  • Initial Load (First GET)
-- Replace XXX with the target page_id
DECLARE  
    client_id VARCHAR2(64) := :APP_USER || ':AppName:XXX';
BEGIN
    IF :APP_PAGE_ID = XXX AND :PXXX_SalaryIn IS NOT NULL THEN 
        xxx_app_ctx_helper_pkg.set_parameter_value(client_id, 'P_SalaryIn', :PXXX_SalaryIn);
    END IF;  
EXCEPTION
    WHEN OTHERS THEN
        apex_application.g_notification := '*** APEX ERROR > DB Session|Init: ' || SQLERRM; 
END;
-- Replace XXX with the target page_id
DECLARE
    client_id VARCHAR2(64) := :APP_USER || ':AppName:XXX';
BEGIN      
    xxx_app_ctx_helper_pkg.clear_ctx(client_id);
EXCEPTION 
    WHEN OTHERS THEN 
        apex_application.g_notification := '*** APEX ERROR > DB Session|Cleanup: ' || SQLERRM;
END;

Seed a default context value on the first page load when items are NULL, so reports have a valid filter before the user submits anything. Place this block before any item-based checks.

-- Place in: Page → Before Header → Process → Execute PL/SQL Code
IF :APP_PAGE_ID = XXX THEN
    DECLARE  
        client_id     VARCHAR2(64) := :APP_USER || ':AppName:XXX';
        defaultSalary NUMBER       := 1000; -- or: SELECT col INTO defaultSalary FROM table
    BEGIN
        IF :PXXX_SalaryIn IS NULL THEN 
            xxx_app_ctx_helper_pkg.set_parameter_value(client_id, 'P_SalaryIn', defaultSalary);
        END IF;  
    EXCEPTION
        WHEN OTHERS THEN
            apex_application.g_notification := '*** APEX ERROR > Init Seed: ' || SQLERRM; 
    END;
END IF;
Alternative Approach

Use server-side conditions to hide reports until a page item is submitted, avoiding the need to seed a default value entirely.

Request Lifecycle​

Each browser interaction is an independent pipeline. Init and Cleanup behave like pre/post middleware wrapping every request.

CyclePhaseWhat Happens
GET (Show)Pre-renderingRuns before APEX builds HTML; same DB session as the render
GET (Show)Post-renderingRuns after HTML is generated, still the same HTTP request
POST (Submit)On SubmitSeparate HTTP request; APEX updates session state, then Submit processes run
POST (Submit)Re-renderFresh GET cycle fires automatically after submit
Key Point

Physical DB sessions can change between requests due to connection pooling, but the APEX session context persists. Treat every request as independent middleware execution — the context values you set in Init will always be fresh.

Verification​

Step 1 — DB Block​

Run directly in SQL Developer or SQL*Plus to confirm the context and package work independently of APEX.

BEGIN
    DBMS_OUTPUT.PUT_LINE('=== BEGIN ==='); 

    xxx_app_ctx_helper_pkg.set_parameter_value('client_id123', 'ATTRIBUTEIN', 'valueIN');
    DBMS_OUTPUT.PUT_LINE(SYS_CONTEXT('TEST_APP_CTX', 'ATTRIBUTEIN'));  
    
    FOR r IN (SELECT namespace, attribute, value FROM GLOBAL_CONTEXT) LOOP
        DBMS_OUTPUT.PUT_LINE(r.namespace || ':' || r.attribute || '=' || r.value);
    END LOOP;
    
    xxx_app_ctx_helper_pkg.clear_ctx('client_id123');
    DBMS_OUTPUT.PUT_LINE('==== END ====');
END;

-- Expected output:
-- === BEGIN ===
-- valueIN
-- TEST_APP_CTX:ATTRIBUTEIN=valueIN
-- ==== END ====

Step 2 — Parameterized View​

CREATE OR REPLACE VIEW Custom_HR_CTX_V AS
SELECT e.*
FROM   hr.employees e
WHERE  e.salary = TO_NUMBER(SYS_CONTEXT('TEST_APP_CTX', 'P_SALARY_IN'));

Use this view in a Classic or Interactive Report region: SELECT * FROM Custom_HR_CTX_V.

Step 3 — Ajax Check​

Filter or sort the IR region. The Initialization PL/SQL fires on each Ajax request, so the context value is always current for the view.

Deep Dive​

Execution Flow​

This is the detailed sequence that ties APEX requests to context-aware SQL. It applies to full page renders (GET), submits (POST), and Ajax refreshes.

  1. Request begins (GET, POST, or Ajax)

    • APEX acquires a pooled DB session for this request.

  2. Initialization PL/SQL runs (per request)

    • Compute a canonical client_id (example: :APP_USER || ':AppName:XXX').
    • If seeding defaults on first load, do it here (or in a Before Header process for the initial GET only).

  3. Set request scope for visibility

    • A Global Application Context is keyed by CLIENT_IDENTIFIER.
    • Ensure the DB session’s CLIENT_IDENTIFIER matches your canonical client_id: 
      DBMS_SESSION.SET_IDENTIFIER(client_id);
  • If you do not want to change APEX’s identifier, use the existing identifier as your key instead: 
    client_id := SYS_CONTEXT('USERENV','CLIENT_IDENTIFIER');

  1. Write context attributes

    • Call the trusted package to write values using DBMS_SESSION.SET_CONTEXT.
    • Keep attribute names consistent across writer and reader. Pick one canonical name and reuse it exactly.

  2. SQL executes

    • Regions, views, and queries run.
    • SYS_CONTEXT('TEST_APP_CTX', '<ATTRIBUTE>') is evaluated and predicates apply.

  3. APEX returns the response

    • HTML for GET, redirect/render for POST, JSON/partial HTML for Ajax.

  4. Cleanup PL/SQL runs (per request)

    • Clear context for that client_id to prevent leakage across pooled sessions.

Performance Notes​

  • SYS_CONTEXT is a SQL function and can be used directly in WHERE clauses and views.
  • Index the filtered table columns normally (for example, hr.employees.salary), then validate the plan as usual with EXPLAIN PLAN (or your standard performance tooling).
  • Treat context values like request-scoped inputs. Keep conversions predictable:
    • If the context value should be numeric, validate before converting (avoid runtime conversion errors).
  • Global Context does not change basic optimizer rules. It behaves like standard predicate evaluation, so measure and tune like any other filter.
